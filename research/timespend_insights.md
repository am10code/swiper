# Time Spend Insights для Swiper

## 1) Текущее состояние (по коду и документации)

### Что уже есть и работает

- У задачи есть фактическое время: `totalTime` (в секундах) и история фокус-сессий `pomodoroSessions`.
- Фокус-время сохраняется из полной карточки задачи через помодоро-таймер; длительность сессии фиксируется и суммируется в `totalTime`.
- Есть поведенческий сигнал откладывания: `postponeCount` и `lastPostponed`.
- Есть секция "Часто откладываемые" (по порогу `postponeThreshold`, по умолчанию 5), сортировка по убыванию `postponeCount`.
- В ФЛОУ есть стабильный "снимок" очереди (`flowSessionOrderedIds`) на один проход, затем переснимок.
- Экспорт/импорт задач уже переносит `totalTime`, `pomodoroSessions`, `postponeCount`.

### Что отсутствует как продуктовая возможность

- Нет явной оценки времени до старта задачи (плановой длительности).
- Нет сравнения план/факт ни на карточке, ни в списках, ни в аналитике.
- В ФЛОУ нет time-aware ранжирования: длинные и короткие задачи не различаются как отдельный фактор.
- Нет сценария контекста доступного времени: "есть 15 минут, предложи релевантные задачи".
- Переносы и свайп-решения не используются для управления очередью задач (только отображаются в отдельных местах).

### Ключевое ограничение текущей логики ФЛОУ

ФЛОУ приоритизирует по важности/дедлайну и частично по рангу приоритизации, но не учитывает:

1. прогноз длительности задачи;
2. размер текущего временного окна пользователя;
3. риск избегания (многократные переносы/свайпы влево);
4. стоимость переключения контекста (start-stop задач).

Итог: система может подсовывать "формально важную", но нерелевантную текущему окну задачу.

## 2) Продуктовые боли и риски

## Боль A: нет "оценка до старта"
- Пользователь не калибрует объём заранее.
- Нельзя понять, где системная недооценка.
- Нельзя строить прогноз завершения дня.

## Боль B: нет "план vs факт"
- Нельзя улучшать качество планирования.
- Нет обратной связи "я стабильно ошибаюсь на +40%".
- Нельзя объяснить, почему задачи "висят".

## Боль C: ФЛОУ не учитывает доступное время
- При окне 10-15 минут ФЛОУ может выдать тяжёлую задачу.
- Растёт фрустрация и количество откладываний.

## Боль D: переносы и свайпы недоиспользованы
- Переносы уже собираются, но не влияют на приоритет выдачи.
- Нет маркера избегаемых задач (важно для управленческих решений).

## 3) Целевая модель данных (time-aware)

Ниже поля, которые дают максимум ценности при умеренной сложности внедрения.

### 3.0 Статусная модель готовности (решение)

Принята модель из 2 статусов:

- `draft` (черновик)
- `ready` (готово к работе)

Смысл:
- `draft` — задача в проработке, может быть неполной.
- `ready` — задача допускается в "production очередь" и в основной ФЛОУ выполнения.

Критерии перевода `draft -> ready`:
- обязательные поля заполнены (минимум: название, приоритет, дедлайн/без-дедлайна осознанно, базовый time budget/оценка или явный `estimateMode = "none"` для вариативных задач);
- декомпозиция сделана по необходимости (не для каждой задачи обязательно, но для сложных/длинных — обязательна);
- нет блокирующих неопределенностей ("непонятно, с чего начать").

Критерии "нужна декомпозиция":
- оценка задачи крупная (например `estimateMode = epic` или диапазон с верхней границей > 120-180 минут);
- высокая неопределенность (`timeConfidence = low`);
- исторически по похожим задачам сильный перерасход.

### 3.1 Новые поля задачи (MVP)

- `timeEstimateMin: number | null`  
  Плановая оценка фокус-времени в минутах (до старта). Пример: 15, 30, 90, 180, 360.

- `timeEstimateUpdatedAt: number | null`  
  Когда оценка обновлялась в последний раз.

- `actualFocusSeconds: number`  
  Явный "источник истины" по фокус-времени (может дублировать `totalTime` на период миграции).

- `timeVarianceRatio: number | null`  
  `(actualMinutes / estimateMinutes)` для задач с оценкой.

- `timeConfidence: "low" | "medium" | "high"`  
  Насколько пользователь уверен в оценке.

- `estimateMode: "fixed" | "range" | "epic" | "none"`  
  Тип оценки: фикс, диапазон, крупная задача на несколько дней, или без оценки.

- `timeEstimateMinRange: { min: number, max: number } | null`  
  Диапазон для задач с высокой неопределенностью (например 90-240 минут).

### 3.2 Поведенческие сигналы (этап 2)

- `flowSkipCount` — сколько раз задача пропущена в ФЛОУ.
- `flowOpenedCount` — сколько раз задача попадала в ФЛОУ-контекст.
- `avoidanceScore` — агрегат из переносов и пропусков.
- `lastFlowDecisionAt` — последний момент решения в ФЛОУ.

### 3.3 Контекст сессии (не в задаче, а в настройках/состоянии)

- `activeTimeWindowMin` — текущее окно пользователя (например 15 минут).
- `flowStrategy` — режим выдачи (`balanced`, `quickWins`, `deepWork`).

### 3.4 Регулярные задачи с переменной длительностью

- Для регулярных задач оценка по умолчанию не обязательна (`estimateMode = "none"` допустим как штатный сценарий).
- Вместо "точной оценки задачи" используется дневной бюджет категории/паттерна:
  - `dailyTimeBudgetMin: number | null` (например, "почта: 30-60 минут в день");
  - `dailyTimeSpentSeconds` (факт за текущий день по этой регулярной задаче).
- Для таких задач в аналитике считается не классический plan/fact по задаче, а:
  - соблюдение дневного бюджета;
  - тренд перерасхода/недорасхода по дням недели.

### 3.5 Эпики (решение по структуре)

Принято решение:
- отдельный список `Epics`;
- в основную "production очередь" попадают не эпики, а их рабочие подзадачи.

Правило:
- эпик выступает контейнером цели и горизонта;
- единицей ежедневного исполнения является подзадача эпика (`ready` + оценка/бюджет);
- если у эпика нет подзадач, он не должен конкурировать в общей очереди с обычными задачами.

## 4) План/факт: что показывать пользователю

### На карточке задачи

- План: `Оценка: 25 мин`.
- Факт: `Потрачено: 42 мин`.
- Отклонение: `+68%` (или `-20%`).
- Подсказка:
  - при сильном перерасходе: "Похоже, задачу стоит разбить на шаги";
  - при сильном недорасходе: "Оценка обычно завышена — можно смелее в планировании".

### В списках/ФЛОУ (компактно)

- Бейдж длительности: `~15м`, `~45м`, `~2ч`.
- Индикатор fit к окну: `fit 15m` / `too long`.
- Индикатор риска избегания: `откладывается часто`.

### В недельной аналитике

- Accuracy планирования: медиана `plan/actual` по задачам с оценкой.
- Распределение задач по корзинам: `<15м`, `15-45м`, `45-120м`, `120м+`.
- Top категорий с перерасходом времени.

## 5) Новый алгоритм ФЛОУ с учётом времени

## Цель
Выбирать следующую задачу не только по важности, но и по шансу реально продвинуть поток в текущем окне времени.

## 5.1 Строгий критерий "пропуска" задачи

Задача считается "пропущенной" в ФЛОУ только если одновременно выполнены оба условия:

1. пользователь явно выбрал действие пропуска (например, Next/Skip в ФЛОУ) без завершения задачи;
2. в текущем открытии карточки не было зафиксировано meaningful progress.

Что считается meaningful progress (и отменяет факт пропуска):
- добавлена ручная запись в лог;
- добавлен/изменен/закрыт шаг декомпозиции;
- запущена фокус-сессия (даже если короткая, например >= 60 сек);
- изменены существенные поля задачи (дедлайн, приоритет, ссылка, регулярность, оценка времени);
- выполнено действие, которое меняет состояние задачи по смыслу ("Выполнено на сегодня", "завершить", "перенести с явным решением даты").

Что не считается meaningful progress:
- открыть/закрыть карточку без изменений;
- просто скролл/переключение секций;
- случайные клики без сохранения изменений.

Уточнение по статусам:
- если задача в `draft`, переход "дальше" в режиме доработки черновиков не считается пропуском производственной задачи;
- `flowSkipCount` учитывается только для задач в статусе `ready` и только в рабочем ФЛОУ.

## Предложенный scoring (MVP)

`finalScore = priorityScore + urgencyScore + fitScore + avoidanceScore + agingBoost`

Где:

- `priorityScore` — существующий `priorityRank`/срочность.
- `urgencyScore` — близость дедлайна.
- `fitScore` — соответствие `timeEstimateMin` текущему окну (`activeTimeWindowMin`).
- `avoidanceScore` — штраф/буст на основе `postponeCount` (+ позже `flowSkipCount`).
- `agingBoost` — мягкий рост веса "залежавшихся" задач, чтобы не было вечных хвостов.
- `epicPenalty` — мягкий штраф для длинных задач при коротком окне (но без полного исключения).

## Режимы выбора в ФЛОУ

- `balanced` (по умолчанию): компромисс важности и времени.
- `quickWins`: сильный вес `fitScore` для коротких окон (5-20 минут).
- `deepWork`: при окне 60+ минут можно поднимать длинные важные задачи.

## Правило "у меня есть 15 минут"

1. Пользователь выбирает окно (быстрый чип в хедере ФЛОУ: 5/15/30/60).
2. Очередь пересчитывается с повышенным весом `fitScore`.
3. Задачи без оценки:
   - либо с нейтральным fit;
   - либо в конце списка (опция стратегии).
4. При завершении/пропуске — локальный пересчёт следующей позиции.

## 5.2 Длинные задачи (2-3 дня и более)

- Не ограничивать модель верхней границей 120 минут.
- Для длинных задач использовать `estimateMode = "epic"` или диапазон (например 240-720 минут суммарно).
- В коротких окнах таким задачам не давать "первый слот", если нет выделенного режима `deepWork`.
- Для ФЛОУ по умолчанию предлагать входной подшаг длинной задачи на 15-30 минут (если декомпозиция есть), а не весь эпик целиком.
- В аналитике длинные задачи считать отдельно от "коротких/оперативных", чтобы не искажать общий показатель план/факт.

## 5.3 Два режима ФЛОУ

Рекомендуемая логика:

1. `Flow Prepare` — отдельный поток доработки `draft` задач (заполнить поля, решить нужна ли декомпозиция, выставить бюджет/оценку, сделать задачу `ready`).
2. `Flow Execute` — поток исполнения только `ready` задач (именно здесь действует score, fit к окну и skip-метрики).

Плюс:
- не смешиваются подготовка и исполнение;
- меньше "ложных пропусков";
- выше предсказуемость производственной очереди.

## 5.4 Capacity дня и дедлайн "Сегодня" (оценка идеи)

Идея: в настройках ввести дневной capacity (например, 240-360 минут фокус-времени) и учитывать его при назначении дедлайна "Сегодня".

Потенциальная ценность:
- защищает день от перегрузки;
- делает план реалистичнее;
- снижает число переносов "по факту не влезло".

Риски:
- жесткий запрет может мешать в реальной жизни (всегда бывают форс-мажоры);
- без override пользователи будут обходить систему.

Рекомендация для продукта:
- делать не hard-block по умолчанию, а `warn + override`:
  - показывать: "Сегодня уже запланировано 320 из 300 минут";
  - предлагать ближайший день с доступным capacity;
  - оставлять кнопку "Все равно поставить на сегодня" с причиной (опционально).

Когда нужен hard-block:
- только как отдельная настройка для дисциплинированного режима планирования.

## 6) Роадмап внедрения

## Этап 1 (MVP, минимальный риск)

1. Ввести статусы `draft` / `ready` и чек готовности перед попаданием в production очередь.
2. Добавить оценку времени/бюджет на задачу (`timeEstimateMin`, `estimateMode`) в карточку и импорт/экспорт.
3. Добавить план/факт в карточку (без сложной аналитики).
4. Добавить переключатель "окно времени" для ФЛОУ.
5. Включить time-aware сортировку ФЛОУ (через `fitScore` и `postponeCount`).
6. Ввести событие `meaningfulProgress` и считать `flowSkipCount` только при отсутствии прогресса.

Ожидаемый эффект:
- быстрый practical gain без сильной перестройки архитектуры.

## Этап 2 (поведенческий интеллект)

1. Учитывать `flowSkipCount` и решения пользователя в ФЛОУ.
2. Ввести `avoidanceScore` и отдельные рекомендации по "залипающим" задачам.
3. Добавить недельный отчёт по план/факт.
4. Добавить поддержку `estimateMode` (`fixed`/`range`/`epic`/`none`) и особые правила для длинных задач.
5. Для регулярных задач включить дневной бюджет времени вместо жесткой оценки на задачу.
6. Внедрить `Flow Prepare` как отдельный режим доведения `draft` до `ready`.

## Этап 3 (оптимизация потока)

1. Политики автодекомпозиции: если задача системно >120 минут, предлагать разбить.
2. Персонализация весов scoring на основе пользовательской истории.
3. Эксперименты стратегий выдачи (A/B-подход внутри локального профиля).

## 7) Ценность vs сложность (для выбора старта)

| Инициатива | Ценность | Сложность | Рекомендация |
|---|---|---|---|
| Статусы `draft/ready` + критерии готовности | Очень высокая | Средняя | Делать сразу |
| `timeEstimateMin` + `estimateMode` в карточке | Высокая | Низкая/средняя | Делать сразу |
| План/факт в карточке | Высокая | Низкая/средняя | Делать сразу |
| Окно времени 5/15/30/60 | Высокая | Средняя | Делать сразу после MVP-полей |
| Time-aware ФЛОУ scoring | Очень высокая | Средняя | Делать в MVP |
| Capacity дня (warn + override) | Высокая | Средняя | Делать в MVP/раннем этапе 2 |
| Поведенческие сигналы skip/avoidance | Высокая | Средняя | Этап 2 |
| Недельная аналитика | Средняя/высокая | Средняя | Этап 2 |
| Дневной бюджет для регулярных задач | Высокая | Средняя | Этап 2 |
| Персонализация весов | Высокая | Высокая | Этап 3 |

## 8) Совместимость данных и импорт/экспорт

### Backward compatibility

- Новые поля должны быть опциональными.
- Для старых задач:
  - `status = ready` для обратной совместимости (чтобы не ломать текущие очереди);
  - `timeEstimateMin = null`;
  - `estimateMode = "none"` по умолчанию;
  - `actualFocusSeconds = totalTime` (или вычисляется при чтении);
  - коэффициенты план/факт только если есть оценка.

### Изменения в импорте/экспорте

- Расширить нормализацию в импорте:
  - валидировать `timeEstimateMin` (целое число > 0, upper bound не зажимать искусственно в 120);
  - валидировать `estimateMode` и диапазоны `timeEstimateMinRange`;
  - безопасно принимать отсутствие новых полей.
- Экспортировать новые поля без нарушения старого формата.

## 9) KPI успеха

- Доля задач с заданной оценкой: целевой рост до 60%+ активных задач.
- Доля задач, корректно проходящих `draft -> ready` до попадания в production очередь: >=80%.
- Median absolute error по план/факт: снижение минимум на 20% за 4-6 недель.
- Доля "fit-сессий" (задача уложилась в выбранное окно): рост минимум на 25%.
- Снижение повторных откладываний (`postponeCount` для top-20 задач): минус 15-25%.
- Рост завершений в коротких окнах (<=15 минут): +20%+.
- Снижение "ложных пропусков" (skip без учета meaningful progress): до <5% от всех skip-событий.
- Для регулярных задач: доля дней в пределах дневного бюджета >=70%.
- Перегрузка "Сегодня" (план > capacity) снижается минимум на 30%.

---

## 10) Anti-goals (чего не делать на старте)

- Не строить тяжёлую ML-модель на первом этапе.
- Не форсировать оценку для регулярных задач с высокой вариативностью (почта, входящие, triage).
- Не усложнять ввод оценки (one-tap пресеты + опция range/epic без перегруза формы).
- Не смешивать в одном потоке доработку `draft` и исполнение `ready` задач.
- Не ломать текущие флоу при отсутствии оценки (всё должно деградировать мягко).

---

## 11) Рекомендуемый стартовый пакет (практически)

Если делать только самое окупаемое:

1. Статусы `draft/ready` + чек готовности (включая "декомпозиция по необходимости").
2. `timeEstimateMin` + `estimateMode` + быстрые пресеты оценки/бюджета на карточке.
3. План/факт в карточке (простая метрика отклонения).
4. Переключатель "есть X минут" в ФЛОУ + настройка дневного capacity с `warn + override`.
5. Изменённая сортировка ФЛОУ с `fitScore` + `postponeCount` + корректный критерий skip.

Это даст наибольший прирост управляемости времени при минимальном риске для текущей архитектуры.
