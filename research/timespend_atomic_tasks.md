# Atomic Implementation Tasks: Time Spend / Readiness / Flow

Документ составлен на основе `research/timespend_insights.md`.  
Приоритет: от быстрых низковисящих задач к сложной многоэтапной перестройке.

## P0 — Низковисящие фрукты (быстро, высокий эффект)

### T01. Добавить поле статуса задачи `draft|ready` в модель данных
- **Тип изменений:** Logic
- **Где править:**
  - `popup/storage.js`
  - `popup/modules/import-export.js`
- **Что изменить:**
  - В `StorageManager.addTask()` добавить `status` (по умолчанию `ready` для обратной совместимости существующего UX).
  - В `updateTask()`/инициализации недостающих полей выставлять fallback `status = 'ready'`.
  - В импорте: принимать `status`, валидировать только `draft|ready`, при отсутствии ставить `ready`.
  - В экспорте: убедиться, что `status` сохраняется.
- **Критерий готовности:**
  - Новые задачи получают `status`.
  - Старые данные загружаются без ошибок.
  - Импорт/экспорт не ломает backward compatibility.

### T02. Добавить time-поля задачи (миграционно-безопасно)
- **Тип изменений:** Logic
- **Где править:**
  - `popup/storage.js`
  - `popup/modules/import-export.js`
- **Что изменить:**
  - Добавить поля: `timeEstimateMin`, `timeEstimateUpdatedAt`, `actualFocusSeconds`, `estimateMode`, `timeEstimateMinRange`, `timeConfidence`.
  - Миграция на чтении: если поля отсутствуют, выставлять безопасные default.
  - `actualFocusSeconds` синхронизировать с `totalTime` для старых задач (лениво при чтении/обновлении).
  - В импорте валидировать `estimateMode` и диапазон `timeEstimateMinRange`.
- **Критерий готовности:**
  - Все поля присутствуют у новых и старых задач после первого update.
  - Импорт старых файлов проходит без регрессий.

### T03. UI-индикация статуса `draft/ready` в списке задач
- **Тип изменений:** UI
- **Где править:**
  - `popup/popup.js`
  - `popup/popup.css`
- **Что изменить:**
  - В рендере карточек добавить бейдж статуса (`Черновик` / `Готово`).
  - Визуально отделить `draft` (нейтральный стиль, без “боевого” акцента).
- **Критерий готовности:**
  - В каждой карточке виден текущий статус.
  - Визуально статусы различимы и не ломают текущую верстку.

### T04. Редактирование оценки времени в карточке задачи
- **Тип изменений:** UI+Logic
- **Где править:**
  - `main.html`
  - `popup/popup.html`
  - `popup/task-card.js`
  - `popup/popup.css`
- **Что изменить:**
  - В блоке редактирования карточки добавить:
    - `estimateMode` (`fixed|range|epic|none`);
    - пресеты минут (5/15/30/45/60/90/120/180/240/360);
    - для `range` поля `min/max`;
    - `timeConfidence`.
  - Сохранять поля через `storage.updateTask(...)`.
- **Критерий готовности:**
  - Пользователь может задать/изменить оценку за 1-2 клика.
  - `estimateMode=none` работает без обязательных полей времени.

### T05. План/факт в карточке задачи
- **Тип изменений:** UI+Logic
- **Где править:**
  - `popup/task-card.js`
  - `popup/popup.css`
- **Что изменить:**
  - Показать блок:
    - План (из оценки),
    - Факт (из `actualFocusSeconds`),
    - отклонение (%) при наличии плана.
  - Подсказки при сильном перерасходе/недорасходе.
- **Критерий готовности:**
  - На задачах с оценкой видно план/факт и отклонение.
  - На задачах без оценки блок не шумит и корректно деградирует.

### T06. Учет `actualFocusSeconds` при завершении помодоро
- **Тип изменений:** Logic
- **Где править:**
  - `popup/task-card.js`
  - `popup/storage.js`
- **Что изменить:**
  - При записи помодоро-сессии обновлять и `totalTime`, и `actualFocusSeconds`.
  - Для legacy-данных оставить совместимость.
- **Критерий готовности:**
  - Факт-время в новой метрике растет синхронно с помодоро.
  - История сессий остается валидной.

## P1 — Средняя сложность, основной value для потока

### T07. Чек готовности `draft -> ready` в карточке
- **Тип изменений:** UI+Logic
- **Где править:**
  - `popup/task-card.js`
  - `popup/popup.css`
- **Что изменить:**
  - Добавить кнопку/переключатель “Пометить как готово к работе”.
  - Перед переводом в `ready` запускать валидатор критериев готовности:
    - заполнены ключевые поля,
    - для сложных задач есть декомпозиция,
    - нет критических пустот.
  - Если критерии не выполнены — показать конкретный список, чего не хватает.
- **Критерий готовности:**
  - Нельзя перевести в `ready` при провале критериев.
  - Пользователь получает понятный список недостающих пунктов.

### T08. Фильтрация production-очереди: только `ready`
- **Тип изменений:** Logic
- **Где править:**
  - `popup/popup.js`
  - `popup/task-card.js`
- **Что изменить:**
  - В активных рабочих списках и в ФЛОУ исполнения использовать только `status='ready'`.
  - `draft` не должен участвовать в боевом ранжировании.
- **Критерий готовности:**
  - `draft` не попадают в production-очередь.
  - Текущая сортировка для `ready` сохраняет прежние инварианты.

### T09. Настройка окна времени пользователя (5/15/30/60/120)
- **Тип изменений:** UI+Logic
- **Где править:**
  - `main.html`
  - `popup/popup.js`
  - `popup/storage.js`
  - `popup/popup.css`
- **Что изменить:**
  - В ФЛОУ добавить переключатель окна времени.
  - Сохранять выбранное окно в `settings`.
  - Пробросить окно в ранжирование ФЛОУ.
- **Критерий готовности:**
  - Смена окна немедленно влияет на очередь.
  - Значение окна сохраняется между сессиями.

### T10. Time-aware score для ФЛОУ (`fitScore + epicPenalty`)
- **Тип изменений:** Logic
- **Где править:**
  - `popup/task-card.js` (функции построения flow-порядка)
  - при необходимости `popup/popup.js` (инициализация/переснимок)
- **Что изменить:**
  - Дополнить текущую сортировку коэффициентами:
    - `fitScore` относительно выбранного окна,
    - `epicPenalty` для длинных задач в коротком окне.
  - Для `estimateMode=none` использовать нейтральный fit.
- **Критерий готовности:**
  - При окне 15 минут очередь поднимает короткие `ready` задачи.
  - В режиме `deepWork` длинные задачи получают преимущество.

### T11. Ввести событие `meaningfulProgress` и точный skip-критерий
- **Тип изменений:** Logic
- **Где править:**
  - `popup/task-card.js`
  - `popup/storage.js`
- **Что изменить:**
  - Фиксировать флаг прогресса в рамках одного открытия карточки.
  - Инкрементировать `flowSkipCount` только при явном skip и отсутствии progress.
  - Для `draft` задач в prepare-потоке skip не считать “боевым”.
- **Критерий готовности:**
  - Лог/декомпозиция/редактирование значимых полей отменяют учет skip.
  - “Пустые перелистывания” учитываются как skip только в `ready`-потоке.

### T12. Отображение time-бейджей в списках и ФЛОУ
- **Тип изменений:** UI
- **Где править:**
  - `popup/popup.js`
  - `popup/task-card.js`
  - `popup/popup.css`
- **Что изменить:**
  - Добавить компактные метки: `~15м`, `~2ч`, `fit/too long`.
  - Для `estimateMode=none` показывать нейтральный бейдж (“без оценки”).
- **Критерий готовности:**
  - Визуальные метки читаемы и не перегружают карточку.
  - Нет артефактов в legacy popup и main.

## P2 — Важные продуктовые расширения (системный контроль)

### T13. Настройка дневного capacity в Settings
- **Тип изменений:** UI+Logic
- **Где править:**
  - `main.html`
  - `popup/popup.js`
  - `popup/storage.js`
  - `popup/popup.css`
- **Что изменить:**
  - В секции настроек добавить поле “Дневной capacity (мин)”.
  - Сохранение в `settings` с нормализацией диапазона.
- **Критерий готовности:**
  - Capacity сохраняется и используется в дедлайн-валидации.

### T14. Проверка capacity при установке дедлайна “Сегодня” (`warn + override`)
- **Тип изменений:** UI+Logic
- **Где править:**
  - `popup/popup.js` (создание/редактирование задач)
  - `popup/task-card.js` (быстрые дедлайны, правка в карточке)
  - `popup/modules/dialog-service.js` (диалог предупреждения)
- **Что изменить:**
  - При попытке назначить “Сегодня” считать суммарный план дня.
  - Если capacity превышен: показать предупреждение, предложить ближайшую дату, дать override.
- **Критерий готовности:**
  - Пользователь видит явный риск перегруза дня.
  - Можно осознанно подтвердить override.

### T15. Режим `Flow Prepare` для `draft`
- **Тип изменений:** UI+Logic
- **Где править:**
  - `main.html`
  - `popup/popup.js`
  - `popup/task-card.js`
  - `popup/popup.css`
- **Что изменить:**
  - Добавить отдельный запуск потока подготовки черновиков.
  - В этом режиме карточка фокусируется на доведении до `ready`.
  - При выполнении чеков задача переводится в `ready` и уходит в production поток.
- **Критерий готовности:**
  - Есть явное разделение Prepare vs Execute.
  - В Prepare не искажаются skip-метрики боевого потока.

### T16. Еженедельная аналитика план/факт (базовая)
- **Тип изменений:** UI+Logic
- **Где править:**
  - `main.html` (новый блок в settings или отдельная секция)
  - `popup/popup.js`
  - `popup/storage.js`
  - `popup/popup.css`
- **Что изменить:**
  - Вывести метрики:
    - median error план/факт,
    - распределение по длительности,
    - топ перерасхода по категориям.
- **Критерий готовности:**
  - Метрики считаются на реальных данных и обновляются корректно.

### T17. Регулярные задачи: дневной бюджет вместо жесткой оценки
- **Тип изменений:** UI+Logic
- **Где править:**

  - `popup/popup.js`
  - `popup/popup.css`
- **Что изменить:**
  - Для recurring задач разрешить `estimateMode=none` как норму.
  - Добавить `dailyTimeBudgetMin` и отображение факта за день.

  - По ним виден контроль “бюджет дня vs факт”.

## P3 — Сложная многоэтапная перестройка

### T18. Ввести сущность Epic и отдельный список `Epics`
- **Тип изменений:** UI+Logic
- **Где править:**
  - `popup/storage.js`
  - `main.html`
  - `popup/popup.js`
  - `popup/popup.css`
  - при необходимости `STRUCTURE.md`/`README.md`
- **Что изменить:**
  - Добавить тип задачи/сущности `epic`.
  - Раздел “Эпики” в меню/навигации.
  - Эпик хранит цель и агрегирует прогресс подзадач.
- **Критерий готовности:**
  - Эпики отображаются отдельно от ежедневной очереди.
  - Создание/редактирование эпика не ломает существующие задачи.

### T19. Связь “подзадача -> эпик” и попадание в production-очередь
- **Тип изменений:** UI+Logic
- **Где править:**
  - `popup/storage.js`
  - `popup/task-card.js`
  - `popup/popup.js`
- **Что изменить:**
  - Добавить `epicId` у задач.
  - В production-очередь и ФЛОУ включать только подзадачи эпиков (не сами эпики).
  - В карточке подзадачи показывать связь с эпиком.
- **Критерий готовности:**
  - Эпики не конкурируют с обычными задачами в ежедневном ранжировании.
  - Подзадачи эпиков ранжируются и исполняются как обычные `ready`-задачи.

### T20. Автодекомпозиция/подсказки для длинных задач
- **Тип изменений:** Logic (+ точечный UI)
- **Где править:**
  - `popup/task-card.js`
  - `popup/popup.js`
  - `popup/popup.css` (для подсказок)
- **Что изменить:**
  - На основе `estimateMode=epic`, `timeConfidence=low`, `timeVarianceRatio` показывать рекомендации:
    - “разбить на 2-5 шагов”,
    - “выделить первый 15-30 минутный подшаг”.
- **Критерий готовности:**
  - Для длинных задач система регулярно подталкивает к исполнимому первому шагу.

### T21. Персонализация весов scoring и локальные эксперименты
- **Тип изменений:** Logic
- **Где править:**
  - `popup/storage.js`
  - `popup/task-card.js` (ranking)
  - `popup/popup.js` (настройки)
- **Что изменить:**
  - Добавить настраиваемые веса `priority/urgency/fit/avoidance`.
  - Включить профили стратегий или lightweight A/B в локальном профиле.
- **Критерий готовности:**
  - Поведение очереди можно тонко настраивать без ручной правки кода.
